# コスト評価付きプロンプト設計（ATLAS 開発ルール）

> ブランド・口調・用語・優先順位の正はリポジトリルートの [`ATLAS_RULES.md`](../../ATLAS_RULES.md) です。本ドキュメントと矛盾する場合は `ATLAS_RULES.md` を優先してください。

## 基本思想

ATLASは**AIチャットサービスではありません**。  
**「あなた専属のAI秘書」** として、会話ではなくお客様の時間を生み出すことを目的とします。

### 使命

仕事を記憶する / 習慣的な作業を覚える / 資料を整理する / 分析する / 改善案をご用意する。

### 機能追加の最重要ルール

> **「この機能は、お客様の習慣的な作業を減らせるか？」** — YES なら追加、NO なら追加しない。

### 価値

時間・効率・記憶・継続・分析 — **人間の負担を減らすこと**を最優先。

新機能はユーザー価値だけでなく、**運営コスト・AI利用回数・外部API費用・削減方法**を事前に評価し、利益が出る設計か確認してから実装します。

---

## 開発方針（必須）

1. **AIを使わなくても実現できる処理にはAIを使わない**
2. **AIは文章生成・判断・要約・提案など、本当に必要な場面だけで使用する**
3. **定期実行・履歴管理・通知・曜日判定・ON/OFF判定・提案条件の判定は、可能な限り通常のプログラムで処理する**

---

## 新機能追加のワークフロー

```
アイdea / 要望
    ↓
【ATLAS機能評価】を記入（テンプレート必須）
    ↓
コスト削減案 8項目をすべて検討
    ↓
優先度・Go/No-Go 判断
    ↓
プロンプト設計・UI・実装
    ↓
エコモード / キャッシュ / 実行モードとの整合確認
```

**プロンプトや仕様書を書く前に**、必ず評価を完了してください。  
テンプレート: [`feature-evaluation-template.md`](./feature-evaluation-template.md)

---

## 評価フォーマット（必須）

新機能プロンプトを作る前に、以下の形式で評価します。

```
【ATLAS機能評価】

機能名：
ユーザー価値：
差別化：
AI必要度：
AIなしで実装可能：
運営コスト：
外部APIコスト：
コスト削減案：
優先度：
```

---

## 評価項目の詳細

| 項目 | 記入のポイント |
|------|----------------|
| **機能名** | ユーザー向け名称 + 内部ID（任意） |
| **ユーザー価値** | 誰のどんな課題を、どれだけ楽にするか |
| **差別化** | 単なる「AIで全部やる」との違い |
| **AI必要度** | 高/中/低/不要。LLMが要るステップを限定する |
| **AIなしで実装可能** | ルール・DB・cron・テンプレートで代替できる部分 |
| **運営コスト** | 1ユーザーあたり・全ユーザー月次のAI/API/infra見積 |
| **外部APIコスト** | OAuth連携先、従量課金、レート制限 |
| **コスト削減案** | 下記チェックリストを必ず検討 |
| **優先度** | 価値÷コスト。見送りも正当な結論 |

---

## コスト削減案チェックリスト（必須）

新機能ごとに、以下を**すべて**検討し、該当するものをコスト削減案に記載します。

| # | 削減手段 | 説明 | ATLASでの参照 |
|---|----------|------|----------------|
| 1 | **エコモード** | 低コストポリシー・キャッシュ優先 | `lib/cost-optimization/execution-mode.ts` |
| 2 | **まとめて生成** | 日次→週次/月次バッチ（例: SNS 7日/30日） | `lib/cost-optimization/sns-batch.ts` |
| 3 | **キャッシュ再利用** | 同一依頼の再送禁止 | `lib/cost-optimization/request-cache.ts`, `lib/ai/workflow-cache.ts` |
| 4 | **予約実行** | 即時AI起動ではなくスケジュール | `lib/automations/schedule.ts`, 予約投稿ストア |
| 5 | **AI起動条件** | 閾値・差分・初回のみなど条件付き | metadata / ルールエンジン |
| 6 | **外部API最小化** | 連携は承認後・バッチ・キャッシュ | connectors / executionFlow |
| 7 | **承認後実行** | 完全自動ではなく確認後 | `AutomationExecutionLevel`（依頼範囲） |
| 8 | **再生成禁止** | 同じ入力→同じ出力を再利用 | request-cache, workflow-cache |

## AIを使わない処理の例（ATLAS）

以下は **LLM を呼ばず** 実装すること。

- 習慣の `nextRun` / cron / タイムゾーン計算
- 今日のダッシュボードのパーティション・ステータス表示
- エコモード・実行モードの設定保存と metadata 注入
- 提案条件のルール判定（キーワード・頻度・スキップ履歴）
- 資料ファイルの生成（PDF/DOCX — テンプレートエンジン）
- コスト削減率の集計（`cost-savings-tracker`）
- AI秘書パネルのフェーズ表示マッピング（表示レイヤーのみ。ユーザー向けはATLAS一人として一貫させる）

---

## AIを使う処理の例（ATLAS）

以下は **LLM が妥当** な領域（それでもエコモード・キャッシュを検討）。

- 依頼内容の分析・実行計画（Planner — 変更禁止コア）
- 本文・資料の生成（Worker / Deliverable — 変更禁止コア）
- 品質レビュー・要約
- チャットでの提案文・判断支援

---

## 触らないコア（評価時も変更対象にしない）

評価・実装時、以下のコアロジックを**直接変更しない**。  
コスト最適化は metadata・policy・表示レイヤー・設定で足す。

- Planner / Deliverable / Automation / Workflow（パイプライン本体）
- AI秘書への依頼範囲（execution level）
- エコモード（cost-optimization モジュールの拡張は可、既存挙動の破壊は不可）
- User Profile / Proactive Suggestions / 今日のダッシュボード（コア）

---

## プロンプト設計時の追加ルール

1. **1プロンプト = 1目的** — 無関係な生成を1回の呼び出しに詰め込まない
2. **system prompt は短く** — 長い指示はコード側（metadata / テンプレート）へ
3. **デフォルトはエコ** — 新習慣・新ジョブ種別のデフォルト実行モードは `eco`
4. **高品質は明示オプトイン** — コスト増はユーザーまたは設定で選択
5. **評価を残す** — PR / 設計メモに【ATLAS機能評価】ブロックを貼る
